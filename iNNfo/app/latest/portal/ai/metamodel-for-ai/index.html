A metamodel in iNNfo defines the ‚Äúlanguage‚Äù of a project:

Anatomy of a Relationship Rule

Secci√≥n titulada ¬´Anatomy of a Relationship Rule¬ª

Relationship rules are defined between Types and govern how Items can be linked.

Components:

  1. SourceType: Where the relationship originates.
  2. TargetType: Where the relationship points.
  3. Label: The unique semantic name of the connection (e.g., assigned_to, depends_on).
  4. Widget: The UI editor used in the Matrix View.
  5. Config: Parameters for the widget (e.g., min/max for scales, options for sets).

AI Generation Guidelines

Secci√≥n titulada ¬´AI Generation Guidelines¬ª

When the AI creates or modifies a metamodel:

  1. Identify Types: Start with 2-4 core Types. Avoid over-engineering.
  2. Define Fields: Add data fields that describe the Items (e.g., due_date, budget).
  3. Establish Relationships: Connect Types using semantic labels.
  4. Select Widgets: Match the widget to the nature of the data (e.g., use scale for priority, not text).
  5. Assign Visuals: Always provide a Lucide icon and a thematic color.

Validation Checklist

Secci√≥n titulada ¬´Validation Checklist¬ª
  • All iNNfo Types have meaningful, natural language names.
  • Each Type has an icon and a color.
  • Relationships use the {widget: type, config: {}} syntax.
  • Field names are descriptive and use lowercase/underscores.
  • Cardinality (e.g., "1" --> "0..*") is used to document intent.

Common Mistakes to Avoid

Secci√≥n titulada ¬´Common Mistakes to Avoid¬ª
  1. Terminology Confusion: Don‚Äôt call them ‚ÄúClasses‚Äù or ‚ÄúInstances‚Äù in the final UI output; use Types and Items.
  2. Ambiguous Labels: Use clear verbs for relationships (e.g., manages instead of link1).
  3. Complex Schemas: Keep the initial metamodel simple; users can always add more Fields later.
